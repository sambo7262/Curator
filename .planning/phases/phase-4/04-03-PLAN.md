<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| *arr manualimport JSON → adapter | untrusted *arr response (rejections, file mappings) crosses here; core never reads its keys |
| Curator → *arr / Plex (network) | X-Api-Key / X-Plex-Token auth over synobridge container-name routing |
| Readarr fault → loop | a book-side outage must not escalate into a music-path failure (ARR-02) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | DoS (availability) | readarr import methods | mitigate | Whole-body swallow→safe-default on every import method (ARR-02); a Readarr 5xx/timeout can never raise into the loop |
| T-04-09 | Tampering | lidarr execute_import (wrong files imported) | mitigate | Explicit per-file files[] with resolved trackIds/albumReleaseId + Move; core pre-filters to empty-rejection resources; NEVER DownloadedAlbumsScan (no blind rescan) |
| T-04-10 | Information Disclosure | X-Api-Key / X-Plex-Token | mitigate | Keys in headers/params only, from settings; never logged; Plex token no-ops if absent rather than being echoed |
| T-04-11 | Spoofing/Integrity | verify_imported false positive | mitigate | verify re-queries the *arr and returns False unless the item truly left the wanted list; Readarr verify returns False on fault (forces quarantine, never a fake success) |
| T-04-SC | Tampering | pip installs | accept | No new package (httpx/respx already pinned + approved). No install task. |
</threat_model>
