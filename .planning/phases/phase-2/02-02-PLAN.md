<objective>
Build the persistent spine: a SQLite (WAL) ledger that is the source of truth for "should I act on this gap?", keyed on stable *arr identity. This delivers STATE-01 (lifecycle status persisted, survives restart) and STATE-02 (dedup on `(arr_app, arr_id)` with a status-preserving upsert). The single most load-bearing correctness rule lives here: the `ON CONFLICT` upsert refreshes metadata only and MUST NOT reset `status` — clobbering an already-`imported`/`searching` row that still appears in wanted/cutoff is the #1 STATE-02 pitfall (RESEARCH Pitfall 1), and a test must prove it does not happen.

Purpose: Every later phase writes lifecycle transitions into this table; getting the dedup key + status-preservation + WAL durability right now prevents a Phase 3/4 schema migration. Per the forward-compat directive, the schema also persists canonical identity (`foreign_id`, the MBID/foreign-book id the adapter already receives) as columns-only — no logic this phase — so Phase 3 matching has its anchor ready.
Output: `app/state/db.py`, `app/state/schema.sql`, `app/state/repo.py`, a FastAPI startup migration hook in `app/main.py`, and `app/tests/test_state_repo.py`.
</objective>
