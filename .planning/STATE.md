## Recent Decisions

- **gate.evaluate is eligibility-before-acceptance and selector ratifies the accept set; matching!=selection is structurally proven (03-05)** — `gate.evaluate` filters each candidate through `quality.gate` + `fakeflac.check` BEFORE scoring, so a below-cutoff / fake-FLAC folder is structurally unable to slip past as a good match (it's recorded as `excluded` with its reason, never scored). On accept it snapshots the best-distance accept set (best + 1e-9 epsilon) and hands it to `selector.select`; `recommend()` already declines genuine rec-gap near-ties, so the selector only ever ratifies a single unambiguous winner (and would tie-break a true multi-accept set) — it never re-judges the match. `selector.py` is the LONE reader of `upload_speed`/`free_upload_slots`/`username`, enforced by an attribute-access source grep in test_gate.py (Pitfall 5 / T-03-06). **Corpus reality correction:** two near-identical copies of the SAME album are a rec-gap AMBIGUOUS decline (not an accept), so the plan's multi-accept selector test was replaced with a clear-winner gate test + a dedicated test_selector.py ladder unit — faithful to MATCH-02, no weakened assertion.

- **The firewall `cutoff`/`allowed`/`items` tokens are matched as QUOTED JSON KEYS, not bare words (03-05)** — `core/quality.py` legitimately emits English reason strings ("below cutoff", "cutoff met") and exposes the neutral `Profile.cutoff_rank`/`allowed` attrs, so `ARR_FIELD_NAMES` matches only the quoted-key forms `"cutoff"`/`"allowed"`/`"items"` (plus `qualityProfileId`, `items[`). This targets the *arr JSON-key access (`body.get("cutoff")`, `profile["items"][i]["allowed"]`) and admits the neutral identifiers + prose. Firewall holds: 0 forbidden tokens across all 8 core modules.
